Known Issues and Limitations

This document tracks known limitations, practical workarounds, and recent fixes.

Current Status

The following major items have already been implemented in code:

  • Sentinel-safe register reads via tryGet(word, word*)
  • Sequential lookup cache for linked-list register searches
  • Early frame completion for fixed-length RTU requests (FC01-FC06)
  • Stream baud callback for non-HardwareSerial transports
  • Optional Modbus ASCII mode (setProtocol(ASCII))

Validation Status Snapshot

This snapshot consolidates practical audit findings that were previously tracked in development test logs.

  • Verified working examples in current workflow: 01 through 07
  • Recently updated compile-verified examples pending fresh runtime sweep: 03, 05, 06, 08, 09
  • Manual verification guides are maintained under docs/manual_testing/

Key findings from practical validation:

  • Arduino USB serial auto-reset can cause false startup failures without a startup delay.
  • Serial Monitor conflicts with Modbus traffic when Modbus uses Serial.
  • Core integration path is strong, but industrial advanced paths remain partially validated or unvalidated.

Active Limitations

0. Validation Scope Gaps (Board/Transport/API Coverage)

  • Current automated and manual verification is strongest on USB Serial0 test profiles.
  • Public examples are documented to match that validated profile first, then explain how to move Modbus onto another UART for debug-heavy deployments.
  • RTU validation is broader than ASCII validation; current ASCII evidence is example-driven rather than full function-matrix coverage.
  • The following areas are not yet fully validated in practical hardware tests:
    • Multi-UART board paths (e.g., MEGA Serial1/Serial2/Serial3 for Modbus while Serial0 is used for logs/programming)
    • ESP32 and other non-AVR board-specific UART behavior
    • SoftwareSerial and other non-HardwareSerial stream transports under realistic traffic
    • Static register pool builds and runtime behavior:
      • MODBUS_USE_STATIC_REG_POOL
      • MODBUS_MAX_DIG_REGS
      • MODBUS_MAX_ANA_REGS
    • RS485 DE/RE runtime timing APIs:
      • setTxEnablePin(...)
      • setTxEnableDelays(...)
      • setTxEnableDelaysUs(...)
    • Endianness mode switching beyond the default path:
      • configureEndianness(MODBUS_LITTLE_ENDIAN)
      • configureEndianness(MODBUS_BIG_ENDIAN_SWAPPED)
    • Direct unsupported-function injection checks for onUnknownFunction(...)
    • Direct FC08 diagnostics request automation (mirror-register checks are covered)
    • Atomic API behavior in integrated runtime scenarios:
      • atomicBegin(), atomicEnd(), atomicGet(), atomicSet(), isAtomicLocked()
    • Throughput and latency characterization of the non-blocking slave.run() state machine under heavy traffic
    • Explicit firmware-side getFloat(...) / getLong(...) assertion coverage

Workarounds:

  • Treat these paths as deployment-specific validation items.
  • Create board-targeted smoke tests before release for each hardware target.

0.1 USB Serial0 Auto-Reset on COM Open

  • On common Arduino USB boards, opening the COM port resets the MCU.
  • Immediate Modbus requests can fail while the sketch is rebooting.
  • This was the main cause of false startup failures during Example 04 development testing.

Workarounds:

  • Wait about 2 to 3 seconds after opening the COM port before the first request.
  • Add retry logic in test or master-side automation.
  • Keep this in mind when comparing Modbus Poll behavior against scripted tests.

0.2 Debug Output Conflicts with Serial0 Modbus

  • If slave.setPort(Serial) is used, the USB Serial port is reserved for Modbus traffic.
  • Serial.print() output and an open Serial Monitor can corrupt or block Modbus frames.
  • Single-UART boards cannot safely use USB Modbus traffic and USB debug prints at the same time.

Workarounds:

  • Keep Serial Monitor closed whenever Modbus uses Serial.
  • On MEGA-class boards, move Modbus to Serial1, Serial2, or Serial3 and keep Serial for logging.
  • For Uno-class boards, use LEDs, coils/register mirrors, or temporary alternative test sketches instead of live Serial debug.

1. Linked-List Storage Cost at Scale

  • Register storage still uses linked lists by default.
  • Lookup remains O(n) in the worst case (cache helps common sequential access).
  • Large maps can increase latency and RAM overhead.

Workarounds:

  • Keep register maps compact and grouped.
  • Use static register pool mode for deterministic memory behavior:
    • MODBUS_USE_STATIC_REG_POOL=1
    • MODBUS_MAX_DIG_REGS=<n>
    • MODBUS_MAX_ANA_REGS=<n>

2. RTU Timing Still Requires a Fast loop()

  • Parser behavior is non-blocking and byte-wise.
  • RTU frame boundary detection still depends on silence-window timing.
  • Long blocking calls in loop() can reduce request throughput.

Workarounds:

  • Call slave.run() frequently.
  • Replace delay() with non-blocking timing patterns (millis() based).

3. Generic Stream Does Not Have Standardized Baud Control

  • setBaud() can directly reconfigure only HardwareSerial.
  • Generic Stream transport speed must be configured by user code.

Workaround:

  • Register setStreamBaudHandler(...) when using custom transports and reconfigure the transport in the callback.

Behavior Notes

Implemented vs Validated

  • The source code implements more capability than the current test workflow validates.
  • In particular, support for FC15, FC16, alternate endianness modes, static register pools, and RS485 timing APIs should be treated as implemented features that still require target-specific verification.

FC01-FC06 Request Length Enforcement

  • Fixed-length request validation for FC01-FC06 remains strict by design.
  • This is protocol-correct behavior (8-byte RTU request frame for these function codes).

get() vs tryGet()

  • get() preserves compatibility and returns MODBUS_REG_NOT_FOUND for missing registers.
  • tryGet() should be preferred for unambiguous reads.

Example:

word value = 0;
if (regBank.tryGet(40001, &value)) {
// Safe: register exists and value is valid (including 0xFFFF)
}

Best Practices

  1. Use tryGet() for read paths where missing registers are possible.
  2. Keep loop() non-blocking and call slave.run() as often as possible.
  3. If Modbus uses Serial, keep Serial Monitor closed and avoid Serial.print().
  4. If you need debug logging, route Modbus to a different UART first.
  5. For RS485, configure DE/RE timing and test at target baud rate.
  6. Validate all register addresses against Modicon ranges.

Roadmap Ideas
